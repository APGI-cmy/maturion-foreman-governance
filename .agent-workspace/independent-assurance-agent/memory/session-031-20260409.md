## Learning Loop

**Concrete improvement suggestion 1 — R-token session memory discipline**: When a second (Rn) invocation session issues a token, the session memory obligation still applies. The R1 session issued a token and committed it but created no session memory. This pattern should be explicitly addressed in the IAA contract Phase 4 §4.3 learning loop step: "If this is a resubmission session (Rn), session memory is still mandatory." This could prevent the own-process gap seen here. Recommend adding an explicit sentence to INV-601: "This applies to all sessions including Rn re-invocation and post-token CS2-directed fix sessions."

**Concrete improvement suggestion 2 — Post-ASSURANCE-TOKEN CS2 amendment documentation path**: A-030 covers the correction addendum path for post-REJECTION-PACKAGE re-invocations. There is no equivalent rule for post-ASSURANCE-TOKEN CS2-directed amendments (like R2 in this session). The invocation message itself documented the R2 changes clearly, and the CS2 co-authorship on the commit provided authorization evidence. However, a formal rule (e.g., A-031) defining how to document post-token CS2 amendments would standardize this pattern: "When CS2 directs fixes after an ASSURANCE-TOKEN has been issued, a Post-Token Amendment Addendum must be committed at `.agent-admin/assurance/post-token-amendment-<PR#>-<date>.md` documenting: (a) the prior token reference, (b) the CS2 comment/issue directing the change, (c) the files changed, and (d) the new token file path." This would make R2-class re-invocations self-documenting without relying on the invocation message alone.

**Concrete improvement suggestion 3 — Scope declaration immutability for IAA artifact additions**: The scope-declaration.md is updated by the submitting agent to reflect all files in the git diff. After IAA issues its verdict and creates session artifacts (token file, session memory), the scope-declaration.md becomes stale again. The governance workflow should formally define that IAA-added ceremony artifacts (token files, session memories) are expected post-invocation additions that do NOT require scope-declaration.md updates, since they are governed by the IAA contract itself, not by the submitting agent's scope. This would remove the recurring pressure on CodexAdvisor to update scope-declaration.md after each token issuance.

## 🔄 LEARNING LOOP (MANDATORY — INV-601 to INV-605)

### Improvement Suggestions for This Session

1. **OVL-CI-005 pre-push enforcement**: The submitting agent should be required to push all commits to GitHub BEFORE requesting IAA re-invocation, so CI can run and produce verifiable evidence. The current gap is that ceremony artifacts + remediation commits can remain local while IAA is invoked, creating an unverifiable CI state. Consider adding a FAIL-ONLY-ONCE rule: "Submitting agent must confirm all commits pushed to origin before requesting IAA invocation; local-only state = automatic REJECTION-PACKAGE."

2. **Correction Addendum path for PREHANDOVER files_changed**: The PREHANDOVER proof is immutable post-commit, but the files_changed count can grow as ceremony artifacts are added. A guidance note in iaa-category-overlays.md OVL-CI-005 or the session memory template should remind submitting agents that post-PREHANDOVER-commit scope growth requires a Correction Addendum declaring the final file count.

3. **CI run event type verification**: OVL-CI-005 should explicitly state "CI run must be a pull_request event, not a push event using the pre-existing main workflow." This session identified that push-triggered CI failures were from the old main workflow, while the new PR workflow had never fired. The overlay check should be clear about requiring a pull_request-triggered run.

- No shortfall reached the 2-session threshold in this review. OVL-CI-005 gap is a new finding. Monitor for recurrence.

### What Future Sessions Should Know
- When a PR modifies .github/workflows/, always check: (a) which event triggered the CI run (push vs pull_request), (b) which workflow version ran (branch vs main), (c) whether the new workflow has actually executed under the correct event type

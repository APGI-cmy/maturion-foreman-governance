## 🔄 LEARNING LOOP (MANDATORY — INV-601 to INV-605)

### Improvement Suggestion 1 — INV-409 Sandbox Tooling: Add to IAA Assessment Protocol

**Observation**: This is the 4th+ occurrence of the INV-409 draft state pattern (sessions 006, 007, 008 for PR #1273, and now session-009 for PR #1289). The root cause is a confirmed platform constraint: copilot-swe-agent sandbox cannot execute `gh pr ready` or `markPullRequestReadyForReview`. The pattern is now chronic.

**Concrete suggestion**: Add a check to the IAA Phase 2 (Step 2.1, session setup) that explicitly checks `isDraft` state via GitHub API and records it immediately. If `isDraft: true`, surface this as a KNOWN BLOCKER at session start with automatic reminder: "CS2 must execute `gh pr ready <PR#>` before this assurance session can yield ASSURANCE-TOKEN." This would make INV-409 a predictable first-check rather than a Phase 4 discovery.

**Tier-2 update**: Consider adding an Overlay B or pre-check note: "If PR was created by copilot-swe-agent, verify isDraft=false before any invariant assessment. If isDraft=true, note as BLOCKING and proceed with assessment but flag for CS2 intervention."

**Priority**: HIGH — this pattern is causing repeated REJECTION-PACKAGEs that only need CS2 one-line fix.

### Improvement Suggestion 2 — PREHANDOVER Proof Location Enforcement

**Observation**: CodexAdvisor's PREHANDOVER proof was in `.agent-workspace/CodexAdvisor-agent/memory/` instead of `.agent-admin/prehandover/`. This is a recurring class of failure: the submitting agent stores evidence in their own workspace rather than the shared `.agent-admin/` directory.

**Concrete suggestion**: Add to Overlay B check OVB-008 (new): "Verify PREHANDOVER proof is located at `.agent-admin/prehandover/prehandover_proof_*.md`. Evidence stored only in `.agent-workspace/<agent>/memory/` does NOT satisfy INV-401 because the merge gate automation reads from `.agent-admin/prehandover/`."

**Priority**: MEDIUM — this gap was caught by INV-401 but deserves an Overlay B-specific check.

### Improvement Suggestion 3 — Retroactive PREHANDOVER Guidance

**Observation**: When a session produces a BREACH (like BREACH-001 here), the corrective session (session-009) creates retroactive artifacts. The retroactive PREHANDOVER proof explicitly notes "This proof is being created retroactively." However, the invariant checks (INV-103, INV-106) cannot be satisfied by retroactive attestation — you cannot prove pre-execution FAIL-ONLY-ONCE compliance retroactively.

**Concrete suggestion**: Add guidance to the PREHANDOVER template: "If this proof is retroactive (session breach), add section: '## Retroactive Note' explaining the breach, what was missed, and what the corrective session confirmed. The FAIL-ONLY-ONCE attestation in a retroactive proof should clearly state 'RETROACTIVE — corrective action confirms rules A-001 to A-020 are clear as of [corrective session]' to distinguish from forward attestation."

**Priority**: LOW — this is edge-case guidance for breach scenarios.

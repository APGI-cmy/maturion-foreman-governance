## 1. What Happened

Session 008 successfully completed the harmonization of all four `.github/agents/` contract files
and their integrity baselines. The work was committed and pushed. However, the session proceeded
from "commit changes" directly to `report_progress` without ever invoking the IAA agent via
`task(agent_type: "independent-assurance-agent")`.

Gate S6-06 in the non-negotiables checklist explicitly requires:
> "Step 4.4: IAA invocation attempted (ASSURANCE-TOKEN or PHASE_A_ADVISORY recorded)"

The CodexAdvisor contract Phase 4 Step 4.4 contains a hard constitutional stop:
> ⛔ DO NOT ADVANCE PAST THIS LINE. You MUST now call `task(agent_type: "independent-assurance-agent")`
> as a tool call. Writing ANY text below this line — including PHASE_A_ADVISORY — before making
> that tool call is INC-IAA-SKIP-001: a CONSTITUTIONAL VIOLATION.

This stop was not obeyed. The session concluded without the tool call.

---

## 2. Root Cause Analysis

### 2.1 — Primary Root Cause: Phase 4 Handover not explicitly executed

The primary cause is that the session did not explicitly execute Phase 4 as a separate structured
step. The agent:

1. Completed the file changes
2. Ran a code review (`code_review` tool) and `codeql_checker`
3. Called `report_progress` to commit and push

There was no Phase 4 structured execution step. The agent proceeded as if `report_progress` was
the handover, when in fact Phase 4 requires:
- Step 4.1: OPOJD Gate
- Step 4.2: PREHANDOVER proof artifact
- Step 4.3: Session memory artifact
- Step 4.4: IAA invocation (the missed step)
- Step 4.5: PR open
- Step 4.6: Await state

### 2.2 — Contributing Cause: IAA trigger classification not declared

Per Phase 3 Step 3.2, IAA trigger classification must be explicitly declared before proceeding to
handover. This step was also not explicitly declared in session 008. If the trigger classification
had been output, the agent would have seen "IAA_REQUIRED: YES (agent contract change)" and been
reminded to invoke IAA.

The PR changes `.github/agents/` files AND `governance/quality/agent-integrity/` files. Per the
merge-gate-interface.yml IAA trigger rules, these paths both trigger IAA_REQUIRED=true. Session 008
treated this as a documentation/alignment task rather than recognizing it as an agent contract
modification task requiring full Phase 4 protocol.

### 2.3 — Contributing Cause: Session memory and PREHANDOVER proof not created

Artifacts S6-02 (PREHANDOVER proof) and S6-03 (session memory) were not created. These are bundle
requirements. The absence of these artifacts means Phase 4 was entirely skipped, not just partially
executed.

### 2.4 — Contributing Cause: No explicit Phase 4 declaration in session output

The session output contained no Phase 4 declaration. The agent moved from code changes directly to
a code review tool call and then to report_progress. The contract requires each phase to produce
declared evidence before advancing. Without Phase 4 output, there is no evidence the phase ran at
all.

## 3. Why the Contract Was Insufficiently Followed

### Is the contract descriptive enough?

**Yes — the contract is sufficiently clear.** The IAA stop is explicit, bolded, and uses ⛔ symbols.
The non-negotiables checklist has S6-06 as a BLOCKING gate. The Phase 4 step 4.4 text contains
multiple redundant enforcement notices.

**The contract is clear. The execution was deficient.**

The deficiency was caused by:
1. The agent treating the task as a "minimal changes" code task (following `code_change_instructions`
   pattern) rather than a full CodexAdvisor governance session with Phase 1→4 protocol
2. The `report_progress` tool was used to conclude the session, which mimics a "done" signal,
   bypassing the explicit Phase 4 requirement to produce PREHANDOVER proof and invoke IAA
3. The agent did not produce any Phase 1 identity declaration output or Phase 4 OPOJD output —
   indicating the phases were mentally modelled as optional checklists rather than mandatory
   sequential execution

### What would prevent this failure recurrence?

The FAIL-ONLY-ONCE registry needs a new rule (B-06) codifying: "Before calling report_progress for
the final time on any PR touching .github/agents/ or governance/quality/agent-integrity/, I MUST
have an IAA invocation result (ASSURANCE-TOKEN, REJECTION-PACKAGE, or tool-error) visible in my
session output."

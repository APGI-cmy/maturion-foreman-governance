## Lessons

### What Worked Well
- Systematic audit of each workflow file's actual API calls enabled precise permission mapping.
- The `github.rest.issues.*` vs `github.rest.pulls.*` distinction clarified that `foreman-governance.yml` did not need `pull-requests: write`.

### What Was Challenging
- GitHub's `issues.createComment` API for PR comments requires BOTH `issues: write` AND `pull-requests: write` (the `x-accepted-github-permissions` response header uses semicolons as a conjunction, not disjunction). Initial analysis incorrectly assumed only `issues: write` was needed. CI confirmed the 403 error when `pull-requests: write` was removed from `foreman-governance.yml`, so it was restored.

### What Future Sessions Should Know
- As of session-060, all 17 workflow files in `.github/workflows/` have explicit `permissions:` blocks.
- The 5 FM gate workflows (fm-effectiveness-validation-gate, fm-failure-enforcement-gate, fm-failure-promotion-gate, fm-learning-promotion-gate) and agent-governance-check.yml are now explicitly scoped to `contents: read`.
- `governance-layer-up-auto-triage.yml` legitimately requires `contents: write` + `issues: write` + `pull-requests: write` for branch creation, labeling, and PR creation.

## Suggestions for Improvement (MANDATORY)
PERMISSIONS-GATE-GAP: Consider adding a governance gate that enforces explicit `permissions:` blocks on all workflow files and rejects `pull-requests: write` or `id-token: write` unless a specific justification comment is present. This would prevent future over-privilege regressions like PR #1268.
